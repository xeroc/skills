# Devpost Platform Guide

## 1. Platform Overview

Devpost is the world's leading hackathon platform and developer community, founded in 2011 (originally as ChallengePost). The platform serves as "the home for hackathons" with a mission to inspire developers to build great software. Devpost powers most of the world's software competitions and provides a comprehensive ecosystem for developers to showcase their work, participate in hackathons, and build portfolios.

**Core Functions:**
- Hosting hackathons (both public and private)
- Developer portfolio creation and management
- Project showcasing and discovery
- Community building for developers
- Judging and evaluation tools for competitions

**Key Features:**
- Hackathon registration and submission system
- Personal developer portfolios (accessible at devpost.com/username)
- Project gallery and discovery (devpost.com/software)
- Team formation and management tools
- Built-in judging and evaluation workflows
- Video submission and embedding capabilities
- Integration with GitHub and other development platforms

**Platform Split:**
- **Devpost Public**: Open hackathons and public portfolios
- **Devpost for Teams**: Private internal hackathons for companies

## 4. Submission Process

**For Hackathons:**

**Step 1: Register for Hackathon**
- Find hackathons at: https://devpost.com/hackathons
- Register for specific competitions
- Review rules and requirements before starting

**Step 2: Start Project Submission**
Navigate to hackathon overview page sidebar and choose:
- **"Start project"** - Creates new "Untitled" project
- **"Import from portfolio"** - Use existing project from your portfolio

**Step 3: Complete Required Submission Steps**

**Manage Team:**
- Add teammates via email invitation
- Share invitation link for team members
- Can work solo or with teams

**Project Overview (Required):**
- Project name*
- Project tagline* (elevator pitch)
- Thumbnail image (JPG, PNG, GIF, 5MB max, 3:2 ratio recommended)

**Project Details (Required):**
- Project story* (supports Markdown and LaTeX formatting)
- Built with tags* (must include sponsor tools and technologies used)
- Try it Out links (may be required depending on hackathon)
- Image Gallery (screenshots, logos, charts)
- Video demo link* (YouTube or Vimeo, usually required - check max length requirements)

**Additional Details (Hackathon-Specific):**
- File Upload (35MB max: zip, pdf, word, apk)
- Submitter Type: Individual/Team/Organization*
- Countries of Residence*
- Category selection*
- Code repository URL*
- MIT License agreement*
- Public availability confirmation*
- Explanation of improvements made during hackathon*
- Developer account ID*
- Feedback on required tools

**Step 4: Submit**
- Review all information
- Agree to hackathon terms and conditions
- Click "Submit project"
- Look for green confirmation notification

**Step 5: Verification**
- Projects marked with special icons get Devpost team review
- Submit 1+ week early for baseline eligibility check
- Receive revision requests if needed

**For Portfolio Projects (Non-Hackathon):**

**Direct Portfolio Addition:**
1. Login to your account
2. Go to your portfolio
3. Click "Add a project to your Portfolio"
4. Complete all required fields:
   - Project name and description
   - Technologies used
   - Links to live demo/repository
   - Images and documentation
5. Click "Save"

**Key URLs:**
- Main submission help: https://help.devpost.com/article/122-how-to-enter-a-submission
- Portfolio guidance: https://help.devpost.com/article/116-adding-projects-to-your-portfolio
- Submission steps detail: https://help.devpost.com/article/126-know-your-submission-steps

## 5. Requirements

**Technical Requirements:**
- Publicly accessible project (code repository required for hackathons)
- Working demo or prototype
- Video demonstration (for hackathons) - YouTube or Vimeo hosted
- Documentation in English
- Open source licensing (MIT License often required for hackathons)

**Content Requirements:**
- Clear project description with "elevator pitch" tagline
- Technology stack documentation ("Built with" tags)
- Visual assets (screenshots, logos, diagrams)
- Team member information and contributions
- Demonstration of functionality

**Submission Format Requirements:**
- Project thumbnail: JPG/PNG/GIF, 5MB max, 3:2 ratio recommended
- Video: Must be publicly hosted on YouTube or Vimeo
- File uploads: 35MB maximum (zip, pdf, word, apk formats)
- Images: Support for screenshots, charts, logos in gallery format
- Text: Markdown and LaTeX formatting supported

**Hackathon-Specific Requirements:**
- Must use required sponsor tools/APIs (tagged in "Built with")
- Follow specific judging criteria
- Meet submission deadlines (typically 48-72 hour events)
- Comply with individual hackathon rules and themes
- Provide pre/post development comparison

**Quality Standards:**
- Functional demonstration required
- Clear explanation of problem solved
- Professional presentation materials
- Working links to repository and demo
- Team coordination and communication

**Legal Requirements:**
- Compliance with hackathon terms and conditions
- Intellectual property agreements
- Open source licensing (when required)
- Country of residence declaration
- Consent for public display and judging

## 6. Strategy & Tips

**Portfolio Optimization:**
- **Star your best projects**: You can star up to 4 favorite projects that stay in the top row
- **Chronological advantage**: Projects display newest to oldest by default
- **Use consistent branding**: Maintain visual consistency across project thumbnails
- **Write for recruiters**: Include your portfolio URL (devpost.com/username) on resumes
- **Cross-platform presence**: Tag and showcase diverse technology stacks

**Hackathon Success Factors:**
- **Submit early**: Projects submitted 1+ week early get eligibility review with revision opportunities
- **Follow the brief exactly**: Use all required sponsor tools and APIs
- **Video is crucial**: Most hackathons require videos; invest time in clear demonstrations
- **Tag comprehensively**: Include all technologies, not just sponsor requirements
- **Story structure**: Use prompted headings in project descriptions for better organization

**Discovery and Visibility:**
- **SEO benefits**: Devpost profiles rank well in Google searches for developer names
- **Community engagement**: Active participation in multiple hackathons increases profile visibility
- **Technology tagging**: Proper tagging helps discovery through platform filters
- **Public showcase**: All projects appear in searchable public gallery (devpost.com/software)
- **Cross-referencing**: Projects link back to hackathons and vice versa

**Platform Navigation:**
- **Dual submission paths**: Can start fresh or import existing portfolio projects
- **Draft system**: Projects save as drafts until final submission
- **Edit flexibility**: Can edit projects after submission (until hackathon deadline)
- **Team management**: Easy teammate addition via email or shareable links

**Long-term Platform Value:**
- **Career documentation**: Shows progression and skill development over time
- **Networking**: Connect with other participants and find future teammates
- **Reputation building**: Consistent participation builds credibility in developer community
- **Startup pipeline**: Many successful startups originated as Devpost hackathon projects

**Content Strategy:**
- **Elevator pitch focus**: 20-second understanding rule for project descriptions
- **Visual storytelling**: Use image galleries to show development process
- **Technical depth**: Balance accessibility with technical detail
- **Problem-first approach**: Lead with the problem you're solving

## 8. Common Mistakes / Gotchas

**Submission Deadline Issues:**
- **Don't wait until last minute**: Platform can experience high traffic near deadlines
- **Time zone confusion**: Hackathon deadlines may be in different time zones than your location
- **"Draft" status trap**: Projects remain in "Draft" until you complete ALL steps and click final "Submit"
- **Missing required fields**: Error messages only appear when trying to submit - check requirements early

**Technical Integration Problems:**
- **Video hosting errors**: Videos must be hosted on YouTube or Vimeo - other platforms won't embed
- **Repository access**: Code repos must be publicly accessible during judging period
- **File size limits**: 35MB limit for uploads catches many teams off-guard
- **License conflicts**: Not understanding MIT License requirements for hackathons

**Content and Presentation Errors:**
- **Weak elevator pitch**: Judges spend ~20 seconds per project initially - make it count
- **Missing "Built with" tags**: Forgetting to tag sponsor tools can disqualify submissions
- **Poor video quality**: Audio issues and unclear demonstrations hurt evaluation
- **Inconsistent team info**: Team member details not matching across all forms

**Platform Navigation Confusion:**
- **Portfolio vs. Hackathon projects**: Not understanding the difference between portfolio pieces and competition submissions
- **Edit restrictions**: After deadline passes, hackathon submissions become uneditable (portfolio projects remain editable)
- **Import limitations**: When importing portfolio projects to hackathons, must still meet all competition-specific requirements

**Strategic Missteps:**
- **One-off participation**: Single hackathon participation limits platform value
- **Ignoring community**: Not engaging with other developers or following interesting projects
- **Underselling projects**: Failing to leverage portfolio URL on resumes and LinkedIn
- **Generic descriptions**: Copy-pasting project descriptions without customization

**Quality Control Issues:**
- **Broken links**: Demo links that don't work during judging period
- **Incomplete documentation**: Assuming judges understand technical context without explanation
- **Team coordination gaps**: Team members submitting conflicting information
- **Post-hackathon neglect**: Not updating projects with improvements made after competitions

**Hackathon-Specific Traps:**
- **Theme misalignment**: Building cool technology that doesn't address hackathon theme/challenges
- **Sponsor tool minimalism**: Using sponsor tools as afterthoughts rather than core features
- **Scope creep**: Attempting projects too large for hackathon timeframe
- **Demo failure**: Not having backup plans when live demonstrations fail

**Legal and Compliance Oversights:**
- **IP assumptions**: Not understanding intellectual property implications of open source requirements
- **Team agreements**: Not clarifying ownership and contribution before starting
- **Geographic restrictions**: Some hackathons have location-based eligibility requirements
- **Age requirements**: Student hackathons may have age or enrollment status requirements
