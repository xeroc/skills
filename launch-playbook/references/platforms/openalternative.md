# OpenAlternative Platform Guide

## 1. Platform Overview

**OpenAlternative** is a community-driven directory of open-source alternatives to proprietary software and applications. Founded by Piotr Kulpinski as a weekend project, it has quickly become the go-to destination for discovering open-source software alternatives.

- **URL**: [openalternative.co](https://openalternative.co)
- **Category**: Developer Tools / Open Source Directory
- **Founded**: 2024 (gained massive early traction - 100k unique visitors in first week, #1 on Hacker News)
- **Creator**: Piotr Kulpinski (also created Dirstarter, EuroAlternative, DevSuite, OpenAds)
- **Mission**: "Be your first stop when searching for open source services"

## 2. Key Stats

- **Traffic**: 100,000+ unique visitors in the first week of launch
- **Community**: 13,000+ tech professionals subscribed
- **Recognition**: #1 position on Hacker News during launch
- **Growth**: Over 1 million users have replaced proprietary tools using the platform
- **Repository**: 1,100+ GitHub stars for the main repository
- **Supported Platforms**: GitHub, GitLab, Codeberg, and Bitbucket repositories
- **Pricing**: Free to submit and list projects

## 3. Who Should Use This

**Ideal for:**
- Open-source software projects seeking visibility
- Developer tools and infrastructure projects
- SaaS alternatives (self-hosted solutions)
- AI/ML platforms and frameworks
- Business software alternatives
- Privacy-focused applications
- Projects with active GitHub/GitLab repositories

**Best fits:**
- Projects with clear proprietary alternatives (e.g., "open-source Slack alternative")
- Actively maintained repositories with recent commits
- Projects with established communities (stars, forks, contributors)
- Self-hosted or on-premise solutions
- Developer-focused tools and platforms

**Not ideal for:**
- Proprietary software
- Abandoned or inactive projects
- Projects without clear repository hosting
- Consumer apps without business/developer focus

## 4. Submission Process

### Step-by-Step Submission:

1. **Create Account**: Visit [openalternative.co/submit](https://openalternative.co/submit)
2. **Sign In Required**: You need to create an account to access the submission form
3. **Fill Submission Form**: Complete the project details including:
   - Project name and description
   - Repository URL (GitHub, GitLab, Codeberg, or Bitbucket)
   - Website URL (if applicable)
   - Category selection
   - Proprietary alternative it replaces
4. **Review Process**: Submit and wait for manual review by the team
5. **Approval**: Projects that meet criteria are automatically added to the directory

**Alternative Methods:**
- **GitHub Contribution**: Fork the [OpenAlternative repository](https://github.com/piotrkulpinski/openalternative) and submit a pull request
- **Database Seed Files**: Add your project via the admin interface or database seed files if contributing directly

## 5. Requirements

### Essential Requirements:
- **Open Source License**: Project must be genuinely open source
- **Active Repository**: Recent commits and ongoing maintenance
- **Clear Purpose**: Must be an alternative to existing proprietary software
- **Repository Hosting**: GitHub, GitLab, Codeberg, or Bitbucket
- **Working Project**: Not just ideas or early-stage concepts

### Recommended Assets:
- **Logo**: High-quality project logo/icon
- **Screenshots**: Visual representation of the software
- **Documentation**: Clear README with installation/usage instructions
- **License File**: Proper open source license
- **Description**: Concise explanation of what the project does
- **Website**: Optional but recommended for credibility

### Technical Requirements:
- Repository must be publicly accessible
- Clear project structure and documentation
- Active community (contributors, stars, forks)
- Regular commit activity (not abandoned)

## 6. Strategy & Tips

### Maximize Visibility:
- **Clear Positioning**: Explicitly state what proprietary software you're replacing
- **Active Maintenance**: Keep your repository updated with regular commits
- **Community Building**: Encourage stars, forks, and contributions
- **Documentation**: Maintain excellent README and documentation
- **Website**: Having a dedicated project website increases credibility

### Optimal Timing:
- Submit after initial development and documentation are solid
- Ideal to have some community traction (stars, forks) before submitting
- Consider submitting around major releases or announcements
- Best when you have bandwidth to respond to increased interest

### Ranking Factors:
OpenAlternative uses an algorithm considering:
- **Repository Stars & Forks** (weighted equally)
- **Project Age** (newer projects get slight boost)
- **Recent Activity** (penalty for inactive projects)
- **Community Engagement** (contributors, issues, discussions)

## 7. Tips & Tricks from the Community

Based on available community insights:

### From Founders & Users:
- **Quality over Quantity**: Focus on being genuinely useful rather than feature-complete
- **Clear Value Proposition**: Make it obvious why someone would choose your alternative
- **Active Engagement**: Respond to GitHub issues and community questions promptly
- **Cross-Promotion**: List on multiple directories simultaneously for maximum exposure
- **Sponsor Consideration**: Notable companies like Notesnook report 10-20% traffic boost from sponsorship

### Community Feedback:
- Search functionality could be improved - make your project easily discoverable with clear keywords
- Users often discover tools through category browsing rather than search
- Having a clear "What it replaces" statement is crucial for discovery

## 8. Common Mistakes / Gotchas

### Submission Rejections:
- **Inactive Projects**: Projects without recent commits get penalized
- **Unclear Purpose**: Not clearly stating what proprietary software you're replacing
- **Poor Documentation**: Lack of clear installation/usage instructions
- **Not Actually Open Source**: Some projects claim to be open source but aren't truly
- **Too Early Stage**: Submitting before the project is actually usable

### Performance Issues:
- **Abandoned After Submission**: Not maintaining the project after listing
- **No Website**: Projects without dedicated websites may get less visibility
- **Poor Repository Presentation**: Unclear README, no license, messy code structure
- **Ignoring Community**: Not responding to increased interest/issues after listing

## 9. Pre-Launch Checklist

- [ ] Clear, comprehensive README.md
